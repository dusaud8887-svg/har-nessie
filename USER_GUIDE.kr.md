# User Guide

한국어 · [English](./USER_GUIDE.md)

## Har-Nessie를 가장 짧게 이해하면

프로젝트 폴더와 목표를 넣으면, 그걸 실행 가능한 run으로 바꾸고, 문서와 현재 저장소 상태를 함께 읽고, 다음 run까지 문맥을 이어 주는 도구입니다.

중요한 점은 Har-Nessie가 단순 모델 래퍼가 아니라 `하네스`라는 것입니다.
즉 run 전, run 중, run 후 상태를 계속 보이게 만드는 쪽에 무게를 둡니다.

## 처음 쓰는 사람에게 권장하는 순서

1. `harness.cmd` 또는 `./harness.sh`를 실행합니다.
2. 로컬 주소를 브라우저에서 엽니다.
3. `새 프로젝트`를 누릅니다.
4. 기존 저장소나 문서가 있다면 `프로젝트 분석`을 먼저 누릅니다.
5. 추천 문서와 첫 run 초안을 확인합니다.
6. `권장: 프로젝트 + 첫 작업 생성`을 누릅니다.

기존 코드베이스로 시작할 때 가장 쉬운 경로는 이 흐름입니다.

## 프로젝트와 run의 차이

- `프로젝트`
  - 오래 유지되는 작업 상자
  - 기본값, 단계, 이어받을 작업, 메모리를 같이 들고 갑니다
- `run`
  - 프로젝트 안에서 실제로 한 번 진행하는 작업 묶음
  - intake, 구현, 복구, 정리 점검 같은 목적을 가질 수 있습니다

나중에도 이어질 일이라면 프로젝트는 새로 만들지 말고 계속 같은 프로젝트를 쓰는 편이 맞습니다.

## intake run은 무엇인가

`*-intake` run은 구현을 바로 시작하는 run이 아니라, 시작점을 맞추는 run입니다.

보통 이런 걸 정리합니다.

- 무엇부터 할지
- 어떤 문서를 기준으로 볼지
- 완료를 무엇으로 볼지
- 문서도 같이 갱신할지

즉, intake에서 질문이 먼저 나오는 건 정상입니다.

## 확인 질문은 어떻게 답하면 되나

짧고 평이하게 답하면 됩니다.

보통 아래 세 가지만 들어가면 충분합니다.

- 원하는 것
- 원하지 않는 것
- 문서도 같이 바꿀지

예:

`권장안으로 진행하고, 범위는 좁게 유지해 주세요. 문서도 같이 갱신해 주세요.`

## run 개요는 어떻게 보면 되나

상단 카드 네 개를 먼저 보면 됩니다.

- `Autonomy Score`
  - 얼마나 맡겨도 되는 상태인지
- `Proof of Work`
  - 실제로 무엇이 검증됐는지
- `Context Engine`
  - 프로젝트 문맥이 얼마나 잘 붙어 있는지
- `Recovery Loop`
  - 복구가 필요한 상태인지

많이 읽기 어렵다면 일단 이 카드들부터 보는 편이 좋습니다.
하네스 관점도 사실 이 네 가지로 요약됩니다. 신뢰도, 근거, 문맥, 복구입니다.

## 중간 운영과 이어가기

프로젝트가 계속 이어질 때는:

- 먼저 `권장 다음 작업 초안`을 써 보고
- 문서가 많이 바뀌었으면 `재분석 후 첫 작업 초안`을 쓰고
- `장기 운영 체크`에서 문서 drift, 반복 실패, 연속성 경고를 보고
- 새로운 단계로 넘어갈 때 `다음 단계 / 재진입`을 씁니다

## 승인 화면은 어떻게 보면 되나

대부분은 아래 세 가지만 확인하면 충분합니다.

- 목표가 맞는지
- 변경 금지 영역을 지키는지
- 첫 작업이 말이 되는지

계획이 명확히 범위를 벗어나거나 위험할 때만 `계획 다시 조정`을 누르면 됩니다.

## 기본 설정은 얼마나 손대야 하나

일반적인 로컬 사용이라면 기본값으로도 충분합니다.

- 계획 담당: 설치된 CLI 중 하나 선택
- 구현 담당: 같거나 다른 CLI — 섞어 쓸 수 있습니다
- 실행 프로필: `즉시 진행`
- Codex 모델: 특별히 더 빠른 작업자가 필요하지 않다면 `GPT-5.4` 유지
- Codex fast mode: 속도가 중요하면 켜 두고, 기본 service tier가 필요할 때만 끔
- 화면 언어: `English`
- AI 응답 언어: `English`

## 언제 재분석하면 되나

아래 중 하나면 재분석할 만합니다.

- 문서가 많이 바뀜
- 프로젝트 방향이 바뀜
- 현재 초안이 현실과 안 맞음
- 최신 저장소 상태 기준으로 다시 intake 하고 싶음

## 언제 사람이 직접 멈춰서 봐야 하나

아래 상황이면 자동화에 바로 맡기지 말고 한 번 더 확인하는 편이 좋습니다.

- 자동화 신뢰도가 `사람 확인 필수`
- 변경 금지 영역이 비어 있음
- 저장소가 dirty 상태라 worktree 격리를 못 씀
- 계획 범위가 목표보다 넓어 보임
- 같은 식으로 두 번 이상 실패함
